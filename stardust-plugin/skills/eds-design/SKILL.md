# Design System Generation

Derive the production-ready EDS CSS design system from approved designs and brand tokens.

## MANDATORY PREPARATION

1. Check that `stardust/brand-profile.json` exists. If not, tell the user to run `/stardust:brand` first.
2. Check that designs exist in `stardust/designs/*.html`. If not, tell the user to run `/stardust:design` first — designs must be approved before the design system is generated.
3. Read `.impeccable.md` for design context. If it doesn't exist, tell the user to run `/stardust:brand` first (it creates this).

---

## Core Principle: Derive from Wireframes, Start from Scratch

**Do NOT patch the AEM boilerplate `styles.css`.** The boilerplate contains arbitrary default values (margins, padding, button sizes, heading scales) that have no connection to any brand. Patching it means some rules get updated while others silently survive, creating a mismatch between the design system and the rendered output.

Instead, generate `styles.css` from three inputs:
1. **The aem.js structural contract** — ~20 lines of CSS that the EDS framework requires (FOUC prevention, header/footer visibility, block loading states). These class names are generated by `aem.js` and `scripts.js` decoration and must be styled.
2. **The approved designs** — the designer's visual judgment calls on type scale, spacing, proportions, button sizing, and visual weight. Extract these values from the design `:root` CSS block.
3. **The brand profile** — colors, font families, voice, and identity from `brand-profile.json`.

Nothing from the boilerplate survives unexamined. If a CSS rule exists in the output, it's either required by aem.js or derived from the designs and brand.

### The aem.js Structural Contract

These rules are required by the EDS framework decoration logic. They handle progressive loading, FOUC prevention, and block status. Include them verbatim:

```css
/* --- aem.js framework contract — do not modify --- */
body {
  display: none;
}

body.appear {
  display: block;
}

header {
  height: var(--nav-height);
}

header .header,
footer .footer {
  visibility: hidden;
}

header .header[data-block-status="loaded"],
footer .footer[data-block-status="loaded"] {
  visibility: visible;
}
```

### Class names generated by aem.js/scripts.js decoration

The following classes are added to the DOM by the decoration pipeline. Your CSS must define visual rules for them — but the values come from the brand, not the boilerplate:

| Class | Generated by | Purpose |
|-------|-------------|---------|
| `.section` | `decorateSections()` | Every `main > div` gets this |
| `.default-content-wrapper` | `decorateSections()` | Wraps non-block content in sections |
| `.button-wrapper` | `decorateButtons()` | `<p>` parent of decorated links |
| `a.button` | `decorateButtons()` | Links inside `<strong>`, `<em>`, or standalone `<p>` |
| `a.button.primary` | `decorateButtons()` | Links inside `<strong>` |
| `a.button.secondary` | `decorateButtons()` | Links inside `<em>` |
| `a.button.accent` | `decorateButtons()` | Links inside `<strong><em>` |
| `.icon` | `decorateIcons()` | Span wrapping SVG icons |
| `{block}-wrapper` | `decorateBlock()` | Wrapper div around each block |
| `{block}-container` | `decorateBlock()` | Section containing a block |

## Phase 0: Extract Design Values from Wireframes

Before generating any CSS, extract the designer's visual decisions from the approved designs.

1. Read all designs in `stardust/designs/*.html`
2. Parse the embedded `<style>` block in each design, specifically the `:root` CSS custom properties
3. Extract these values (the design is rendered at 1440px desktop, so these are desktop-resolution values):
   - **Heading font sizes** — `--heading-xxl`, `--heading-xl`, `--heading-lg`, etc.
   - **Body font sizes** — `--body`, `--body-sm`, etc.
   - **Spacing** — `--spacing-xs` through `--spacing-2xl`, `--section-padding`
   - **Content max-width** — `.contained` or similar max-width value
   - **Button styles** — padding, font-size, font-weight, letter-spacing, border-radius from `.btn`
   - **Line heights** — heading and body line-heights
4. Cross-reference across designs — if multiple designs define different values for the same token, use the homepage design as the authority
5. Read `stardust/brand-profile.json` for colors and font families

These extracted values become the **desktop-resolution design tokens**. The design is the authority on sizing and spacing because that's where the designer made visual judgments in context.

## Phase 1: Generate CSS

1. Use the design-extracted values (Phase 0) for sizing, spacing, and proportions
2. Use `stardust/brand-profile.json` for colors and font families
3. Use `brand-css-generator` (from eds-site-builder) to generate initial CSS, then verify/override with design values
4. The generated CSS must include:

**In `styles/styles.css`:**

Start from the aem.js structural contract above, then add:

- **CSS custom properties** for all brand colors, typography, spacing, border-radius, and the nav height
- **Font fallback faces** — size-adjusted `local()` fallbacks to reduce CLS (e.g., `lora-fallback` using local Georgia)
- **Responsive type scale** — mobile-first heading sizes with `min-width` breakpoints at 600px, 900px, 1200px
- **Base body styles** — font-family, color, line-height, background from brand tokens
- **Heading styles** (h1–h6) — font-family, weight, size scale, line-height, color, margin from brand
- **Paragraph/list styles** — margin rhythm matching the brand's vertical spacing
- **Link styles** — brand link color, hover treatment, text-decoration rules
- **Button styles** — for `a.button`, `.button.primary`, `.button.secondary`, `.button.accent`:
  - Padding, font-size, font-weight, letter-spacing, border-radius derived from brand
  - Match the proportions and visual weight of the design buttons (compact, precise, not flabby)
  - Hover/focus/disabled states for each variant
- **Section layout** — `main > .section` padding (use brand spacing, typically `--spacing-xl` to `--spacing-2xl` vertically), content wrapper max-width and horizontal padding
- **Image defaults** — `max-width: 100%`, auto height
- **Icon defaults** — size for `.icon` spans

**Critical: section padding must use the same values the design stage uses (typically 80px vertical). If the design renders generous section spacing, the EDS page must match.**

**In `styles/lazy-styles.css`:**

Section style variants applied via `section-metadata`. For each variant (dark, warm, highlight, cta-band, or whatever the brand needs):
- Background color, text color, heading color overrides
- Button inversions (e.g., white button on dark/red backgrounds)
- **Full section padding** — every colored section needs vertical padding (not just margin), so the background extends edge-to-edge with generous breathing room
- Typography adjustments (opacity on body text in dark sections, etc.)
- Eyebrow/label styles if the brand uses them

**In `styles/fonts.css`:**
- @font-face declarations for all brand fonts
- Font-display: swap for performance
- Multiple formats (woff2 primary, woff fallback)

**In `fonts/`:**
- Downloaded web font files (woff2/woff format)
- If brand fonts are Google Fonts, download them (don't use CDN links — EDS performance requirement)
- If brand fonts are commercial/unavailable, select the closest Google Font alternative and flag the substitution

**Google Fonts download strategy (in order of preference):**
1. Use the Google Fonts CSS API with a Chrome user-agent to get woff2 URLs:
   ```
   curl -sH "User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 (KHTML, like Gecko) Chrome/120.0.0.0 Safari/537.36" \
     "https://fonts.googleapis.com/css2?family=Font+Name:wght@400;700&display=swap"
   ```
   Then download the woff2 URLs from the response. Use the latin subset URL (last in the list for each weight).
2. **Do NOT** use `https://fonts.google.com/download?family=` — this endpoint returns invalid/empty zip files.
3. For variable fonts, a single woff2 file covers multiple weights. Use `font-weight: 400 700` range syntax in @font-face.

4. Write `stardust/design-tokens.json` — a structured export of all CSS custom properties for programmatic access:

```json
{
  "colors": {
    "primary": { "property": "--color-primary", "value": "#C8102E" },
    "accent": { "property": "--color-accent", "value": "#333F48" }
  },
  "typography": {
    "headingFamily": { "property": "--heading-font-family", "value": "Sentinel, Georgia, serif" }
  },
  "spacing": {
    "xs": { "property": "--spacing-xs", "value": "8px" }
  }
}
```

### Token Architecture: designs → design-tokens.json → styles.css

The approved designs are the **origin of visual decisions**. The pipeline flows in one direction:

1. **Designs** — branded static HTML rendered at 1440px (desktop). The designer makes visual judgment calls about type scale, spacing, proportions, and visual weight by seeing them in context. The design `:root` CSS block contains the authoritative desktop-resolution values.

2. **design-tokens.json** — derived FROM the design CSS. Records the designer's approved desktop-resolution values as a structured JSON export. This file is the **single source of truth** for all downstream consumers.

3. **styles.css** — consumes design-tokens.json values as CSS custom properties. Since EDS is mobile-first, `styles.css` must define **smaller base values** for mobile, then scale up to the design-tokens.json values at the desktop breakpoint (`min-width: 900px`). The desktop `@media` block **must exactly match** the design-tokens.json values.

**This means:** at desktop viewport, the EDS page and the design produce identical typography, spacing, and proportions — because design-tokens.json was extracted from the design, and styles.css matches design-tokens.json. There is no token drift because the values flow in one direction: design → tokens → CSS.

### Responsive Type Scale

`design-tokens.json` records one value per token (desktop). `styles.css` must derive a mobile-first scale:

| Token | Mobile (base) | Tablet (≥600px) | Desktop (≥900px = design-tokens.json) |
|-------|--------------|-----------------|--------------------------------------|
| `--heading-font-size-xxl` | ~70% of desktop | ~85% of desktop | **design-tokens.json value** |
| `--heading-font-size-xl` | ~70% | ~90% | **design-tokens.json value** |
| `--body-font-size-m` | 2px smaller | — | **design-tokens.json value** |
| `--body-font-size-s` | 2px smaller | — | **design-tokens.json value** |

The exact mobile/tablet ratios are a design judgment, but the desktop values are non-negotiable — they must match design-tokens.json.

### Consistency Verification

After generating `styles.css` and `design-tokens.json`, verify alignment:

- Section vertical padding in `styles.css` must match design `--section-padding` value
- Button padding/font-size/weight must match design `.btn` values
- Heading sizes at desktop breakpoint must match design `:root` heading sizes
- Content max-width must match design `.contained` width
- Body font sizes at desktop breakpoint must match design `:root` body sizes

If any value in the generated CSS doesn't match the design, the design wins — update the CSS. The design is upstream; the design system is downstream.

## Phase 2: Quality Gate (Agent Automatic)

The designer does NOT see intermediate states. Run these checks and fix issues before exposing the result:

1. Run `/impeccable critique` against the dev server (start it if not running: `npx -y @adobe/aem-cli up --no-open`)
   - Check: no banned fonts (Roboto, system-ui as primary, generic sans-serif alone)
   - Check: color contrast meets WCAG AA for all text/background combinations
   - Check: typographic hierarchy is clear (h1 > h2 > h3 visually distinct)
   - Fix any issues found automatically

2. Run `/impeccable typeset` to refine the type hierarchy if needed

3. Run `/impeccable colorize` to verify color application and accessibility

4. Use `/verification` (from superpowers) to check that generated CSS custom properties match brand-profile.json values — catch any drift between the source tokens and generated output

## Phase 3: Expose to Designer

1. The design system is visible through the dev server at http://localhost:3000
2. Tell the designer: "Your design system is applied. Open http://localhost:3000 to see the base styles. The boilerplate content will now use your brand's colors, fonts, and spacing."
3. If the designer wants changes: update brand-profile.json or tweak CSS directly, then re-run the quality gate
4. When satisfied: "Design system complete. Run `/stardust` to see your next step, or `/stardust:eds-build` to start building blocks and pages."

## Note on Block CSS

The design system generates **only global styles** (`styles.css`, `lazy-styles.css`, `fonts.css`). Block CSS files (`blocks/{name}/{name}.css`) are **not** generated here — they are created fresh during the Build stage.

**Block CSS must also start from empty** — not from the boilerplate block CSS. The Build stage writes each block's CSS from scratch using the design tokens defined here. The same principle applies: nothing inherited, everything intentional. See the Build skill for details.

The header and footer blocks are a partial exception: their JS handles fragment loading (structural plumbing worth keeping), but their CSS is rewritten from scratch in the Build stage like every other block.
